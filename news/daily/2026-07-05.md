## What materially matters today

### 1) Better Models: Worse Tools — SOTA models regress on non-standard tool schemas
- Track: ai_change
- Why it matters to me: This is a concrete failure mode in how frontier models interact with external tooling. If you're building agent systems (your own business or anyone's), the assumption that "better model = better tool use" is false. Opus 4.8 and Sonnet 5 are *worse* at arbitrary tool schemas than older siblings because RL was done in a permissive internal harness (Claude Code) that absorbed errors. Models learned a rigid prior for Claude Code's schema and now hallucinate extra fields when faced with anything different.
- Tradeoff / tension: Using the latest SOTA model gives you better general reasoning but worse structured tool fidelity. Enabling `strict` mode fixes the hallucination but increases complexity limits and may degrade performance on other tasks. The tension is between flexibility (permissive harness absorbs errors) and correctness (strict mode rejects bad output but may block valid edge cases).
- Implication: Agent system design must account for schema drift. Any non-trivial tool interface needs either constrained decoding (grammar-aware sampling) or a validation/retry layer. The model's RL environment is invisible to you but determines its real-world behavior. If you're building on top of Claude or GPT agents, assume tool-call schemas will need explicit constraint enforcement — don't rely on the model to respect arbitrary JSON structures.
- Action: Watch — audit your own agent tool schemas for this failure mode. If you're using Claude for tool calls, test with strict mode enabled and measure the tradeoff.
- Confidence: high
- Source: Armin Ronacher (lucumr.pocoo.org) — https://lucumr.pocoo.org/2026/7/4/better-models-worse-tools/

### 2) GPT-5.5 Codex: reasoning-token clustering at fixed intervals correlates with degraded performance
- Track: better_decisions
- Why it matters to me: Multiple independent researchers confirmed that GPT-5.5 in Codex clusters reasoning tokens at 516, 1034, 1552 (step 518) — a precise arithmetic sequence that looks like hard truncation or routing to a cheaper tier. Responses terminating at exactly 516 tokens are frequently wrong on complex tasks. Adding explicit "extend reasoning" instructions bypasses it. This is empirical evidence of a hidden inference optimization degrading user-facing quality.
- Tradeoff / tension: OpenAI is trading accuracy for throughput/cost at quantized reasoning blocks. Users on paid plans ($100-$200/month) get systematically worse performance on complex tasks without visibility into what happened. The workaround (prompt engineering to force longer reasoning) shifts the burden to the user and doesn't solve the underlying routing issue.
- Implication: Model benchmarks and version numbers don't capture the full picture. The same model name can have hidden degradation modes that only emerge under telemetry. For your own tooling decisions, monitor your own usage patterns rather than relying on vendor claims. If you're evaluating GPT-5.5 for production agent work, this clustering behavior on complex tasks is a real risk. Consider sticking with GPT-5.4 or adding retry/extension logic.
- Action: Think — if you're using GPT-5.5 in Codex for complex tasks, add a reasoning-extension directive to your prompts or benchmark your own outputs. Don't assume newer = better on reasoning-heavy workloads.
- Confidence: high
- Source: GitHub issue #30364 (openai/codex) — https://github.com/openai/codex/issues/30364
