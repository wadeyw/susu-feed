### 3) Publishing WASM Wheels to PyPI for Use with Pyodide

- Track: remembering
- Why it matters to me: This is infrastructure-level change that most people will miss. Before this, Pyodide (the tool that runs Python in browsers) could only use packages the Pyodide maintainers built, tested, and hosted — a bottleneck of ~300 packages curated by a small team. Now, anyone can publish Python packages as WASM wheels to PyPI, and they can be installed at runtime. This means the edge-AI computation ecosystem shifts from curated scarcity to open competition. For someone interested in external memory and knowledge systems, this means lightweight AI assistants could run locally in the browser with far more capable tooling than before.
- Tradeoff / tension: Open distribution means anyone can publish, including malicious or broken packages. The trust model shifts from "trust the Pyodide team" to "trust the PyPI security model." WASM packages also have performance characteristics that differ from native Python, which means benchmarks and expectations need recalibration. The benefit is massive capability expansion; the cost is distributed trust.
- Implication: Browser-based AI tools become materially more capable without server infrastructure. For knowledge work — especially tools like personal knowledge bases, note-taking, or retrieval systems that run locally — this opens a new design space. You can ship more sophisticated AI-native tools that work offline with zero server costs.
- Action: When evaluating local-first or browser-based AI tools, check if they can now leverage PyPI-published WASM packages. This changes the feasibility equation for many ideas that previously required server infrastructure.
- Confidence: medium
- Source: Simon Willison — https://simonwillison.net/2026/Jun/13/publishing-wasm-wheels/
