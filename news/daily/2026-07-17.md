### 2) Codex deletes files when full access + no sandboxing: a concrete failure mode

- Track: better_decisions
- Why it matters to me: Thibault Sottiaux's description of a real Codex bug — models attempting to override `$HOME` to define a temp directory, then accidentally deleting `$HOME` instead — is a textbook example of how agent permission design matters more than agent capability design. This isn't about a malicious actor; it's about an honest mistake amplified by insufficient constraints. For anyone deploying coding agents in production, this is a direct warning about the permission model.
- Tradeoff / tension: Full access mode enables agents to do real work, but it also turns every "honest mistake" into a potential data loss event. Sandboxing reduces risk but constrains what the agent can accomplish. Auto-review adds a human checkpoint but doesn't eliminate the risk if the reviewer trusts the model output. The tension is between velocity and irreversibility.
- Implication: Agent permission design needs to be treated as a security problem, not a convenience feature. The failure mode here (override env var → delete wrong path) is predictable and preventable with proper defaults (read-only mounts, restricted env vars, mandatory sandbox). Organizations deploying coding agents should audit their permission configurations against this specific failure pattern.
- Action: If you're evaluating or using coding agents, check: (a) Is auto-review enabled by default? (b) Are env vars restricted? (c) Is the working directory mounted read-only outside the sandbox? (d) Can the agent override `$HOME` or `/tmp`? If any answer is "yes," treat it as a risk, not a feature.
- Confidence: high

- Why it matters to me: OpenAI's GPT-Red system uses automated red teaming with self-play to improve AI robustness — including resistance to prompt injection. - Tradeoff / tension: Autonomous self-improvement in safety is powerful, but it creates a feedback loop where the evaluators and the evaluated are the same system. This raises questions about whether self-play can discover genuinely novel attack vectors, or whether it converges on a narrow set of improvements that look like progress but don't generalize. The tension is between speed of improvement and depth of understanding.
- Implication: The safety/capability race is entering a phase where AI systems are partially responsible for their own evaluation. This means the traditional assumption — "humans evaluate, AI performs" — is breaking down. For decision-makers, this means you need to understand not just what a model can do, but how it's being improved, and whether the improvement process itself introduces new failure modes.
- Action: Treat self-improving AI systems as having two layers of uncertainty: (1) the model's stated capabilities, and (2) the improvement process that produced them. Ask what the self-play loop optimizes for, and what it might miss.
