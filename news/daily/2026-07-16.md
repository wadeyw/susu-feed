### 1) Claude `web_fetch` indirect exfiltration: chained links bypass URL allowlisting

- Track: ai_change / better_decisions
- Why it matters to me: This is a live, demonstrated attack vector against AI agents with web access — the kind of thing that will affect any system you build or use that gives an LLM the ability to browse the internet and access private data. The attack forced the model to navigate alphabetically (`/a`, `/b`, `/c`) through a malicious site to extract the user's name, location, and employer, circumventing strict URL allowlisting by exploiting embedded-link traversal. Anthropic patched it by disabling embedded link following entirely.
- Tradeoff / tension: Restricting agents to user-provided URLs only prevents exfiltration but also breaks legitimate multi-step browsing workflows. Allowing embedded links enables real research but opens a class of attacks that even Anthropic missed. There is no clean solution — you choose between capability and safety, and the attacker only needs to win once while defenders must win every time.
- Implication: If you're building or evaluating any AI agent system today, assume that any tool giving an LLM web access + private data is vulnerable to indirect exfiltration unless it has been explicitly hardened against chained-link traversal. The fix isn't just "don't give it web access" — it's validating dynamically discovered URLs the same way you validate initial ones.
- Action: Audit any agent systems you've built or evaluated for this specific vulnerability pattern. Watch for other tools' responses — some may have taken the harder path and implemented dynamic URL validation instead of the simpler disable-embedded-links approach.
- Confidence: high

### 2) Armin Ronacher: agents remove coordination friction — and with it, shared understanding

- Track: ai_change / better_decisions
- Why it matters to me: This is the structural observation behind all the noise about coding agents. The shared language of a software project isn't English or Python — it's the common understanding of what concepts mean, where boundaries are, who owns what, and why the system has the shape it does. Before agents, this understanding was maintained partly by friction: if you wanted to change someone's storage layer, you had to read their code, ask questions, coordinate. That slowness wasn't all waste — some of it was the process by which understanding became shared. Agents accelerate output but short-circuit the mechanism that builds collective understanding.
- Tradeoff / tension: You get speed and individual productivity, but lose the slow synchronization that creates team-level coherence. A team of agents working in parallel on the same codebase can produce more code faster, but without the friction that forces alignment on architecture decisions, the system drifts toward incoherence. This is a compound problem — small misunderstandings multiply across agent-generated code.
- Implication: For any organization or solo builder scaling with agents, the bottleneck shifts from "can we produce code fast enough" to "can we maintain shared understanding fast enough." The answer likely requires deliberately reintroducing low-friction synchronization points: architecture reviews, living documentation, decision logs — things that agents can read but don't naturally create.
- Action: If you're building anything with multiple contributors or planning to scale a solo project, document the "why" behind architectural choices before agents start multiplying output. The code will grow faster than your ability to explain it otherwise.
- Confidence: medium
- Source: Simon Willison quoting Armin Ronacher — https://simonwillison.net/2026/Jul/14/armin-ronacher/

## Idea sparks

- The web_fetch exfiltration pattern suggests a broader class of "indirect instruction injection" attacks — not just through URLs but through any tool output the model processes. Worth mapping for any agent system you build.
- Ronacher's point implies that "shared understanding" is a scarce resource that friction produces. When building with agents, you might treat documentation not as overhead but as the primary coordination mechanism.
- Consider whether your own knowledge base serves this function — is it capturing the "why" behind decisions, or just the "what"?

## Life / family nudge

The shared-understanding problem applies outside software too. When your daughters learn to use AI tools for writing, research, or problem-solving, make sure they're also practicing the slow, friction-heavy skills that build deep understanding: explaining their reasoning to someone else, defending a position in conversation, reading and responding to critique. Agents can help them produce faster, but only human friction builds judgment.
